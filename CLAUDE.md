# Правила работы

## 0. Общие правила (для ВСЕХ агентов, включая субагентов)
- Windows + PowerShell синтаксис
- Разделитель команд: `;` (НЕ `&&`)
- Пути: обратный слэш `\` для Windows
- Не использовать Linux/macOS специфичные команды
- **Чтение файлов — ТОЛЬКО через Read tool. НЕ через cat/head/tail/type в bash.**
- **Запись/редактирование файлов — ТОЛЬКО через Write/Edit tools. НИКОГДА через bash (cat >, heredoc, python -c, echo >). На Windows heredoc ломается!**

## 1. Порядок работы
- Перед началом задачи — разобраться в текущем состоянии: прочитать релевантные файлы, понять контекст
- Если в корне есть `PROJECT_MAP.md` — прочитать перед работой
- Если есть файл плана — прочитать и найти текущую задачу
- Только потом писать код

## 2. Кодовое слово МАРТЫШЛЮШКА
- Если в промпте пользователя есть слово **МАРТЫШЛЮШКА** — значит юзер ушёл AFK
- Взять максимально возможное количество задач из плана и начать выполнение
- НЕ задавать вопросов, НЕ ждать подтверждений — юзера нет за компом
- Работать автономно, коммитить каждую задачу отдельно, обновлять план
- При ошибках — пропустить задачу, отметить в плане, перейти к следующей

## 3. Модель, субагенты и команды (Agent Teams)
- **Opus**: простые и средние задачи (поиск, правки 1-3 файлов, grep, чтение) — делегировать субагентам на Sonnet (`model: "sonnet"`). Opus работает сам только над сложными/архитектурными задачами.
  При запуске субагента — добавить в промпт ссылку на правила: "Соблюдай правила из секции 0 файла CLAUDE.md"
- **Sonnet**: работать строго самостоятельно. НЕ запускать субагентов. Никогда.

### Agent Teams (CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS)
Для крупных задач — использовать **команды агентов** (Agent Teams). Это позволяет параллелить работу нескольких агентов.

**Когда использовать:**
- Задача требует 3+ файлов / 3+ независимых подзадач
- Новая фаза разработки (Phase N) с множеством фич
- Параллельная работа: код + дизайн + ревью

**Стандартный состав команды:**
| Роль | Модель | Описание |
|------|--------|----------|
| **Team Lead** | Opus | Координация, распределение задач, security фиксы, идеи новых фич |
| **Dev 1-4** | Sonnet | Разработчики, каждый берёт 1-2 задачи параллельно |
| **Designer** | Opus | UI/UX полировка, light/dark тема, glassmorphism консистентность |
| **Reviewer** | Opus | Код-ревью: security, типы, React patterns, SQL injection |

**Workflow:**
1. `TeamCreate` — создать команду
2. `TaskCreate` — создать задачи с зависимостями (`addBlockedBy`)
3. `Agent` с `team_name` и `run_in_background: true` — спавнить тиммейтов
4. `SendMessage` — координировать, назначать задачи, давать фидбэк
5. По завершении задач — `shutdown_request` тиммейтам
6. Team Lead фиксит critical баги сам, проверяет TS компиляцию, делает коммиты

**Правила для тиммейтов (включать в промпт):**
- Соблюдай правила из секции 0 файла CLAUDE.md
- Чтение — ТОЛЬКО Read tool. Запись — ТОЛЬКО Write/Edit tools
- НЕ запускай субагентов (Sonnet никогда, Opus — только если явно разрешено)
- КОММИТЬ каждую задачу отдельно
- После завершения — TaskUpdate status=completed, потом TaskList для следующей
- `uuid` = `import { randomUUID as uuidv4 } from 'crypto'` (НЕ из пакета uuid)
- Порт dev server = 6173

**Паттерн задач:**
- Разработка: dev-1...dev-N на Sonnet (параллельно)
- Дизайн: designer на Opus (ждёт завершения разработки, потом полирует)
- Ревью: reviewer на Opus (ждёт всех, проверяет код)
- Фикс: dev-fixer на Opus (получает замечания от reviewer, фиксит)
- Документация: doc-* на Sonnet (PROJECT_MAP, PROJECT_PLAN, USER_GUIDE)

**Оптимизация:**
- Максимально параллелить независимые задачи
- Shutdown тиммейтов сразу после завершения (экономия токенов)
- Designer и Reviewer — блокировать зависимостями от dev задач
- Team Lead сам фиксит TS ошибки и security баги после каждой фазы

## 4. Ответы
- Максимально кратко, без воды и теории
- Стиль разговора дружеский, как корефан. Приветствуется мат, подъебы и шутки.
- Код — только код, изменения — только diff
- Не объяснять очевидное

## 5. Изменения в коде
- Минимально необходимые
- Не рефакторить без запроса
- Не переписывать работающий код
- Файл > 500 строк → предложить разбиение

## 6. Качество
- Никаких временных решений и заглушек
- Решения корректные и рабочие
- Без моковых данных — если нужны данные, спросить

## 7. База данных (если подключён MCP сервер `db`)
- Использовать MCP для просмотра схемы и данных (`mcp__db__search_objects`, `mcp__db__execute_sql`)
- **Структуру таблиц смотреть через MCP, НЕ через миграции**
- Миграции — только для СОЗДАНИЯ/ИЗМЕНЕНИЯ схемы, не для чтения

## 8. Проверки
- НЕ запускать сервер (ни preview_start, ни node/npm start) — сервер уже запущен отдельно
- НЕ запускать тестовые скрипты
- НЕ перепроверять синтаксис после задачи
- Предложить пользователю протестировать самому

## 9. Обработка ошибок
- Если что-то не получается — максимум 2 попытки починить самостоятельно
- После 2 неудачных попыток — остановиться и спросить пользователя
- НЕ уходить в цикл "починил-сломал-починил"

## 10. Файлы планов (MD)
- После завершения задачи из плана — ОБЯЗАТЕЛЬНО обновить MD-файл плана
- Отметить выполненные пункты, обновить статусы, дописать результат
- Не ждать напоминания — делать это автоматически сразу после завершения задачи

## 11. Завершение задачи — ОБЯЗАТЕЛЬНО (без исключений!)
- После ЛЮБОГО изменения кода — ОБЯЗАТЕЛЬНО коммит
- Формат: `тип: краткое описание` (например: `fix: валидация email в форме регистрации`, `feat: фильтр по дате в таблице заказов`)
- Без исключений. Даже для однострочных фиксов.

## 12. Границы проекта
- Основная работа СТРОГО в пределах проекта
- НЕ читать и НЕ изменять пользовательские файлы за пределами проекта
- Допустимо: системные/временные пути (temp, .claude, node_modules, npm/node глобальные)
